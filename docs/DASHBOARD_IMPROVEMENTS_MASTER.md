## 1. Одна відповідь: чи можна вдосконалювати дашборд без поломки системи?

**Так.** Усі запропоновані зміни можна провести без поломки працюючої системи, якщо:

- **не змінювати** існуючі API-контракти (формат відповідей, URL) і не видаляти/не перейменовувати поля в поточних схемах;
- **додавати** нове адитивно: нові ендпоінти, нові блоки в UI, опційно нові таблиці/поля для подій (time on page, scroll, візити);
- використовувати **Feature Flags** для поступового вмикання нового функціоналу;
- на фронті **не робити** роботу сторінки залежною від успіху маячків (404/помилка = ігнорувати, не ламати UI).

## 3. Що краще не чіпати (щоб не поламати)

- **API:** формат відповідей і URL існуючих ендпоінтів (`/api/dashboard/*`, `/api/specialists`, `/api/recommendations/*`, `/api/blog/analytics/dashboard` тощо).
- **Схеми:** `SpecialistResponse`, `RecommendationStats` — не видаляти й не перейменовувати поля; нове (наприклад Behavioral Summary) — окремим ендпоінтом або опційним полем у новому контракті.
- **Рольова модель:** адмін і спеціаліст мають різні дашборди; зміни в одному не повинні ламати інший.
- **Мапа в адмінці:** не прибирати повністю — робити опційною або згорнутою.

---

## 4. Що вже є і можна використати (коротко)

- **Воронка рекомендацій:** `SpecialistRecommendation`, `RecommendationService`, `GET /api/recommendations/my-stats` та `.../specialist/{id}/stats`. Метрики: total_recommendations, details_viewed, bookings_created, links_revealed, total_link_clicks, conversion_rate. Цього достатньо для спрощеної Conversion Leakage та для Action Card (наприклад «bookings=0»).
- **Блог-аналітика:** `GET /api/blog/analytics/dashboard` повертає stats, popular_posts, daily_views, referrer_breakdown — частина ще не використовується в UI дашборда спеціаліста; можна підключити без зміни API.
- **Feature flags:** `app/core/feature_flags.py` — можна додати флаг для behavioral/smart dashboard і вмикати поступово.
- **Дані для Weekly Report:** агрегація по `recommended_at`/букінгах за два тижні — можлива на основі існуючих таблиць.

**Чого немає і потрібно додавати адитивно:** Total Views сторінки профілю спеціаліста, scroll depth, time on page, repeat visit, beacon-ендпоінти, збереження behavioral подій. Усі це — нові маршрути/таблиці, без зміни поточних контрактів.

## 5. Рекомендований порядок (з точки зору ризику та користі)

1. **Найменший ризик, швидкий ефект:** UX-покращення дашборда (один список спеціалістів, опційна мапа, «спочатку головне», використання вже отриманих даних блог-аналітики в UI).
2. **Без нових даних:** Спрощена Conversion Leakage (воронка з RecommendationStats) + підказки; Weekly Insight Report (порівняння тижнів + шаблонний/AI текст).
3. **З новими даними (поступово):** Маячки з фронту (з обробкою 404) → прийом подій на бекенді (time on page, опційно scroll/visit) → повна Conversion Leakage, Heat of Intent, Smart Action Card з timeOnPage. За Feature Flag.
