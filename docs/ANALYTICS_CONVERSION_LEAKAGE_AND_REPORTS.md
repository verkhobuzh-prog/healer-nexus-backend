# Звіт: чи можна впровадити Conversion Leakage, Heat of Intent та Weekly Insight без поломки системи

**Мета:** тільки аналітика. Відповісти: (1) чи можна ці зміни провести, не ламаючи систему; (2) наскільки серйозні зміни; (3) чи щось із цього вже існує.

---

## 1. Що пропонується (коротко)

| Фіча | Суть |
|------|------|
| **Conversion Leakage View** | Воронка втрат: Total Views (100%) → Deep Views (Scroll > 50%) → Engaged (Time > 2 min) → Clicks (соц/контакт) → Bookings. Підказки: провал на Deep Views → "Змініть перший екран"; провал на Clicks → "Офер чіткіший". |
| **Heat of Intent** | Сегменти: Cold (< 30s), Warm (1–2 min, читають але не клікають), Hot (> 2 min + repeat). Мета: показувати, що "контент працює", навіть якщо бронювань мало. |
| **Weekly Insight Report** | Порівняння current_week vs past_week, "human-like" висновок. Приклад: "Профіль на 18% цікавіший (Warm зросла), бронювання на місці. Оновіть слоти в календарі." |

---

## 2. Що вже є в системі (що не потрібно дублювати)

### 2.1. Воронка рекомендацій (частина логіки вже є)

У бекенді вже реалізована **воронка по рекомендаціях**, але вона не збігається повністю з "Conversion Leakage View".

**Модель:** `SpecialistRecommendation` — один запис = одна рекомендація одного спеціаліста одному користувачу.

**Події (як вони збираються зараз):**

- **recommended_at** — коли спеціаліста "рекомендували" (чат або пошук: `record_recommendation` при search або в chat tool).
- **details_viewed / details_viewed_at** — коли користувач переглянув деталі **в контексті чату** (виклик `get_specialist_details` у `chat_tool_executor` → `record_details_viewed`). Це **не** відкриття веб-сторінки профілю.
- **booked / booked_at, booking_id** — запис на консультацію.
- **links_revealed / links_revealed_at** — користувач запросив контакти (після підтвердженого букінгу).
- **link_clicks** (JSON) — кліки по соц/контакт посиланнях.

**API:** `GET /api/recommendations/my-stats` та `GET /api/recommendations/specialist/{id}/stats` повертають агрегати:

- `total_recommendations`, `details_viewed`, `bookings_created`, `links_revealed`, `total_link_clicks`, `click_breakdown`, `conversion_rate`.

**Висновок:** у вас уже є воронка виду **Recommendations (100%) → Details viewed → Links revealed → Link clicks → Bookings**. Це можна використати як основу для "спрощеної" Conversion Leakage (без scroll і без time). Повноцінна Conversion Leakage з "Total Views профілю", "Deep Views (scroll > 50%)" та "Engaged (time > 2 min)" — **ще не існує** (див. нижче).

### 2.2. Перегляди профілю спеціаліста (веб-сторінка)

- Сторінка **`/specialists/{slug_or_id}`** (HTML) **не викликає** жодного трекінгу: ні `record_recommendation`, ні окремого "profile_view". Тобто "Total Views" саме сторінки профілю зараз **немає**.
- "Details viewed" у воронці — це лише факт виклику деталей спеціаліста **в чаті**, не відкриття веб-профілю.

Тому "Total Views (100%)" у сенсі "усі, хто зайшов на картку/профіль" — **не існує**; є лише "кількість рекомендацій" (показали в чаті/пошуку).

### 2.3. Час на сторінці, скрол, повторні візити

- У БД **немає** полів: час на сторінці (time on page), глибина скролу (scroll depth), повторні візити (repeat visit) до профілю спеціаліста.
- Є лише дати подій: `recommended_at`, `details_viewed_at`, `booked_at`, `links_revealed_at`. По ним можна рахувати лише "скільки часу між recommended і details_viewed" тощо, але не "скільки хвилин людина провела на сторінці профілю".

Тому **Heat of Intent** (Cold &lt; 30s, Warm 1–2 min, Hot &gt; 2 min + repeat) **не існує** — для нього потрібні нові дані (час на сторінці, повторні візити).

### 2.4. Щотижневі звіти та "human-like" висновки

- **`app/ai/analytics.py`** — клас `AIAnalytics` з методом `generate_daily_report`: зараз це **заглушка** (фіксовані числа та один текст "Попит на енергопрактики зріс на 20%..."). Реальної логіки порівняння тижнів і генерації тексту **немає**.
- Дані для щотижневого порівняння **можна** отримати з існуючих таблиць: агрегувати `SpecialistRecommendation`, `Booking` по періодах (current_week vs past_week) через `get_specialist_stats(days=7)` за два періоди або окремі запити по датах. Тобто **дані для Weekly Report є**, але сам механізм "current_week vs past_week + human-like текст" **ще не реалізований**.

## 3. По кожній фічі: чи існує, чи можна додати без поломки

### 3.1. Conversion Leakage View (воронка втрат)

| Етап у вашому описі | Є зараз? | Джерело / примітка |
|---------------------|----------|---------------------|
| Total Views (100%)  | Частково | Немає "переглядів сторінки профілю". Є лише **total_recommendations** (показали в чаті/пошуку). Можна використати їх як базу 100%. |
| Deep Views (Scroll > 50%) | Ні | Немає збору scroll depth. Потрібен новий збір (фронт + опційно бекенд). |
| Engaged Views (Time > 2 min) | Ні | Немає збору time on page. Потрібен новий збір. |
| Clicks (Social/Contact) | Так | Є **links_revealed**, **total_link_clicks**, **click_breakdown**. |
| Bookings | Так | Є **bookings_created**. |

**Чи можна провести зміни без поломки:**

- **Так.**  
  - Варіант A (мінімальний): побудувати **спрощену** воронку тільки з існуючих даних: **Recommendations (100%) → Details viewed → Links revealed → Link clicks → Bookings**. Без змін контрактів API — лише новий ендпоінт або новий блок у дашборді, який використовує вже наявний `RecommendationStats` (або аналог). Систему не ламає.  
  - Варіант B (повна версія): додати збір "Total Views профілю", "scroll > 50%", "time on page" (маячки з фронту + нові таблиці/поля). Робити це **адитивно**: нові ендпоінти прийому подій, нові поля/таблиці; існуючі API не змінювати. Ризик поломки — низький, якщо не чіпати поточні схеми та маршрути.

**Наскільки серйозно:**  
Повна версія — середній обсяг: новий збір подій (фронт), прийом на бекенді, схема зберігання. Спрощена версія — невеликий обсяг, можна зробити лише на існуючих даних.

**Чи не пропустили — вже є?**  
Частково: воронка "рекомендації → details viewed → links → bookings" **вже є** в даних і в API. Візуал "Conversion Leakage" з підказками ("змініть перший екран", "офер чіткіший") — **немає**; його можна додати поверх існуючих метрик.

### 3.2. Heat of Intent (Cold / Warm / Hot)

| Сегмент | Є зараз? | Що потрібно |
|---------|----------|-------------|
| Cold (&lt; 30s) | Ні | Час на сторінці/картці. Зараз не збирається. |
| Warm (1–2 min, читають, не клікають) | Ні | Час на сторінці + факт "не клікнув". Зараз немає time. |
| Hot (&gt; 2 min + repeat) | Ні | Час на сторінці + підрахунок повторних візитів. Повторні візити до профілю спеціаліста зараз не визначені (немає окремого поняття "visit" з user/session). |

**Чи можна провести зміни без поломки:**

- **Так.** Сегментація — це логіка поверх даних. Якщо додати збір **time on page** та **session/visit** (або repeat по user_id + specialist_id за період) адитивно (нові таблиці/ендпоінти), існуючу систему не ламаємо. Heat of Intent тоді можна рахувати на бекенді або на фронті (якщо бекенд віддає сирі події/агрегати).
- Якщо **не** додавати time/repeat, можна зробити **дуже спрощену** сегментацію лише по існуючих подіях (наприклад "ні разу не клікав" = Warm-подібно, "клікав/забронював" = Hot-подібно), але це вже не справжній "Cold/Warm/Hot по часу та повтору".

**Наскільки серйозно:**  
Повноцінний Heat of Intent — середній обсяг: збір часу та візитів, схема, агрегація по сегментах. Без цих даних фіча **не існує** і не була пропущена — її треба будувати.

**Чи не пропустили — вже є?**  
Ні. Сегментів по часу та повторних візитах у поточному коді немає.

### 3.3. Weekly Insight Report (retention-двигун)

| Компонент | Є зараз? | Примітка |
|-----------|----------|----------|
| Порівняння current_week vs past_week | Частково | Дані є: можна викликати `get_specialist_stats` за два періоди або агрегувати по `recommended_at`/датах. Готового ендпоінту "тиждень vs тиждень" немає. |
| Human-like висновок | Заглушка | `AIAnalytics.generate_daily_report` — фіксований текст, не генерація з даних. |

**Чи можна провести зміни без поломки:**

- **Так.** Новий ендпоінт типу "GET /api/recommendations/my-stats/weekly-insight" або "GET /api/dashboard/weekly-insight", який: (1) рахує метрики за поточний і попередній тиждень з існуючих таблиць; (2) формує текст (шаблони або AI). Існуючі API не змінюються. Ризик поломки — низький.
- Дані для порівняння вже є; потрібна лише логіка періодів і генерація речення.

**Наскільки серйозно:**  
Невеликий–середній обсяг: один новий ендпоінт, бізнес-логіка порівняння, шаблони або один виклик AI для тексту. Заглушка в `ai/analytics.py` не заважає — можна залишити її або поступово замінити реальною логікою.

**Чи не пропустили — вже є?**  
Частково: **дані** для щотижневого порівняння є; **готового Weekly Insight Report** (тиждень vs тиждень + human-like текст) — немає. Є лише ідея в заглушці daily report.

## 5. Висновок

- **Чи можна такі зміни провести, не ламаючи систему?**  
  **Так.** Усі три фічі можна робити адитивно: нові ендпоінти, нові таблиці/поля для подій (де потрібно), використання існуючих даних (recommendations, details_viewed, bookings, link_clicks). Існуючі контракти й маршрути не змінюються.

- **Наскільки серйозно?**  
  - Найменш серйозно: **спрощена Conversion Leakage** (лише з поточних даних) та **Weekly Insight Report** (дані вже є).  
  - Серйозніше: повна **Conversion Leakage** з scroll/time та **Heat of Intent** — потрібен новий збір подій і схема зберігання.

- **Чи щось вже існує і можна було пропустити?**  
  - **Воронка втрат:** частково так — ланцюжок recommendations → details viewed → links → bookings **вже є**; можна одразу робити спрощену візуалізацію "Conversion Leakage" та підказки без scroll/time.  
  - **Heat of Intent:** ні, не існує; Cold/Warm/Hot по часу та repeat треба будувати з нуля (з новими даними).  
  - **Weekly Report:** дані для порівняння тижнів є; сам звіт і "голос платформи" — ні, є лише заглушка в `ai/analytics.py`.

Код і структуру проєкту в цьому звіті не змінювали — лише аналітика на основі поточного стану репозиторію.
