# Аналітика: чи можливі зміни Smart Layer / Action Card / Behavioral без поломки системи

**Мета:** оцінити **без зміни коду та структури** — чи реалізовані запропоновані зміни (Smart-логіка, таблиця правил пріоритетів 1–5, Behavioral Layer з visibilitychange/heartbeat, маячки, Feature Flags, Behavioral Summary окремо від SpecialistResponse, Frontend Blueprint) **без поломки працюючої системи**. Лише аналітика, висновки та умови безпеки.

---

## 1. Що пропонується (коротко)

- **Smart-логіка:** таблиця правил для Action Card (пріоритети 1–5). Приклад: пріоритет 1 (Red) — `bookings=0` і `timeOnPage>120s` («хоче, але боїться»).
- **Behavioral Layer:** трекінг часу на сторінці через `visibilitychange` та heartbeat **без зовнішніх сервісів** (приватність, швидкість).
- **Data collection:** фронт надсилає «маячки» (beacons); якщо бекенд не готовий — запити можуть давати 404, нічого не падає.
- **Feature Flags:** вмикати шар поступово.
- **Schema compatibility:** не змінювати `SpecialistResponse`, додати Behavioral Summary як **окремий об'єкт**, на фронті merge.
- **Frontend Implementation Blueprint:** документ з джерелами даних, візуальними станами (Red/Orange/Green), логікою accordion для блоків (Hero, Action Card, Insights).

Питання: **чи можливо все це зробити без поломки працюючої системи?**

---

## 2. Що вже є в системі (релевантне)

### 2.1. Воронка рекомендацій і «книги»

- **Модель:** `SpecialistRecommendation` — події: `recommended_at`, `details_viewed` / `details_viewed_at`, `booked` / `booked_at`, `booking_id`, `links_revealed`, `link_clicks` (JSON). Один запис = одна рекомендація одному користувачу.
- **Сервіс:** `RecommendationService` — `record_recommendation`, `record_details_viewed`, `record_booked`, `record_links_revealed`, `record_link_click`; `get_specialist_stats(specialist_id, days)` повертає агреговані метрики.
- **API:**  
  - `GET /api/recommendations/my-stats` (спеціаліст) → `RecommendationStats`.  
  - `GET /api/recommendations/specialist/{specialist_id}/stats` (адмін) → ті самі `RecommendationStats`.
- **Схема `RecommendationStats`:** `specialist_id`, `specialist_name`, `total_recommendations`, `details_viewed`, `bookings_created`, `links_revealed`, `total_link_clicks`, `click_breakdown`, `conversion_rate`.

**Висновок:** умови на кшталт «bookings=0» для спеціаліста вже можна обчислити з існуючих даних (з `RecommendationStats` або з БД). **Змінювати ці API/моделі не обов’язково** — достатньо читати їх і комбінувати з новими метриками.

### 2.2. SpecialistResponse і контракти

- **Де використовується:** `GET /api/specialists`, `GET /api/specialists/{id}`, `POST/PUT` спеціалістів. Схема: id, name, service_type, specialty, hourly_rate, bio, delivery_method, is_verified, is_active, тощо.
- **Висновок:** якщо Behavioral Summary **не додавати в цю схему**, а віддавати окремим ендпоінтом або окремим полем-об’єктом у wrapper’і — існуючі клієнти (мобільні додатки, інші інтеграції) не зламаються. **Сумісність збережена.**

### 2.3. Трекінг подій у поточній системі

- **Блог:** є `record_view` (перегляд поста); при помилці — лог, запит не падає. Патерн «надіслати подію, якщо бекенд не прийняв — не критично» вже використовується.
- **Рекомендації:** події (details_viewed, booked, link_click) викликаються з бекенду (chat_tool_executor, profile_router) після дій користувача. Окремих «beacon»-ендпоінтів з фронту для цих подій немає.
- **Час на сторінці:** у БД **немає** збереженого `timeOnPage` або аналогу (немає таблиці/поля для «час перегляду картки спеціаліста»). Є лише `Conversation.duration_seconds` — для чату, не для сторінки деталей спеціаліста.

### 2.4. Feature flags

- **Модуль:** `app/core/feature_flags.py` — `get_flag(project_id, flag_name)`, `set_flag_local`, `get_all_flags`. Флаги задані по проекту (наприклад `healer_nexus`).
- **Висновок:** додати новий флаг (наприклад `behavioral_dashboard` або `smart_action_cards`) — **адитивна зміна**, існуюча логіка не змінюється. Вмикання/вимикання нового шару без поломки core.

---

## 3. Оцінка по пунктах: чи можливо без поломки

### 3.1. Data collection: маячки (beacons), 404 як «нічого не ламається»

- **Ідея:** фронт відправляє події (наприклад «time_on_page», «heartbeat») на URL; якщо бекенд ще не реалізував прийом — відповідь 404, фронт обробляє (ігнорує або не показує помилку користувачу).
- **Чи можливо без поломки:** **так**.  
  - Поточна система не залежить від цих URL. Якщо ендпоінта немає — клієнт отримує 404, обробка на стороні клієнта (try/catch, перевірка status). Існуючі сторінки й API продовжують працювати.  
  - Якщо пізніше додати ендпоінт (наприклад `POST /api/behavioral/event` або `/api/track/beacon`) — це **новий** маршрут; старі контракти не змінюються.
- **Умова:** фронт не повинен вважати успіх маячка обов’язковим для роботи сторінки (наприклад не блокувати UI при 404). Тоді «бекенд ще не готовий» = безпечно.

### 3.2. Feature Flags: поступове вмикання

- **Ідея:** вмикати Smart Layer / Action Card / Behavioral поступово через флаг.
- **Чи можливо без поломки:** **так**. У проєкті вже є механізм флагів по `project_id`. Додавання нового флага не змінює існуючі виклики; умовне відображення блоків на фронті (або умовна відправка маячків) не впливає на core-бекенд.
- **Умова:** за замовчуванням флаг вимкнений, поки новий функціонал не протестований.

### 3.3. Schema compatibility: Behavioral Summary окремо, без зміни SpecialistResponse

- **Ідея:** не змінювати `SpecialistResponse`; Behavioral Summary — окремий об’єкт, на фронті merge з відповіддю по спеціалісту.
- **Чи можливо без поломки:** **так**.  
  - Варіант A: новий ендпоінт типу `GET /api/specialists/{id}/behavioral-summary` (або `/api/recommendations/specialist/{id}/behavioral-summary`). Існуючі `GET /api/specialists`, `GET /api/specialists/{id}` залишаються без змін.  
  - Варіант B: у майбутньому опційно додати в якийсь **новий** DTO поле `behavioral_summary: Optional[BehavioralSummary]` лише там, де це явно «нова версія» API, а старі клієнти продовжують використовувати поточний контракт.  
  Найбезпечніший варіант — окремий ендпоінт або окремий запит за summary; тоді **SpecialistResponse взагалі не чіпається**.
- **Умова:** не видаляти й не перейменовувати поля в існуючих схемах; не робити обов’язковими нові поля для старих URL.

### 3.4. Матриця тригерів (пріоритети 1–5) і Action Card

- **Ідея:** бізнес-логіка «пріоритет 1: bookings=0 і timeOnPage>120s» тощо; відображення Red/Orange/Green.
- **Чи можливо без поломки:** **так**, за умови:  
  - Обчислення пріоритету використовує **існуючі** дані (recommendations, bookings — вже є в `RecommendationStats` / БД) **плюс** нові (timeOnPage), якщо вони з’являться.  
  - Якщо timeOnPage ще немає — правило «timeOnPage>120s» не виконується (наприклад вважати false); пріоритет визначається лише по bookings/details_viewed. Це не ламає поточну систему, лише обмежує «розумність» до наявних даних.  
  - Логіка може спочатку жити **тільки на фронті** (два запити: specialists/stats + behavioral-summary або recommendations; фронт рахує пріоритет). Бекенд при цьому не змінюється.  
  Або пізніше винести обчислення пріоритету в бекенд (новий ендпоінт/поля) — знову адитивно.

### 3.5. Behavioral Layer: visibilitychange + heartbeat, без зовнішніх сервісів

- **Ідея:** на фронті — visibilitychange + heartbeat для часу на сторінці; дані не в Google Analytics, лише власні маячки.
- **Чи можливо без поломки:** **так**. Це реалізація на клієнті; бекенд не змінюється, поки не з’явиться прийом цих подій. Якщо прийому немає — фронт може слати в void або на URL, що повертає 404 (і коректно це обробляти). Існуюча робота сайту не залежить від цього.

### 3.6. TimeOnPage у бекенді (майбутнє)

- **Поточний стан:** у БД і API **немає** збереження часу на сторінці (спеціаліста/картки). Є тільки події рекомендацій (recommended → details_viewed → booked → links).
- **Щоб реалізувати правило «timeOnPage>120s» на бекенді:** потрібно або нове поле (наприклад в окремій таблиці behavioral_events: specialist_id, user_id/session_id, time_on_page_seconds, created_at), або розширення існуючої моделі. Це **адитивні зміни**: нова таблиця або нові nullable-поля не змінюють існуючі записи і не ламають поточні запити. Існуючі ендпоінти залишаються без змін.
- **Висновок:** збереження timeOnPage можна додати пізніше без поломки, якщо робити це окремими сутностями/ендпоінтами.

### 3.7. Frontend Implementation Blueprint

- **Ідея:** документ з джерелами даних (API + behavioral), візуальними станами (Red/Orange/Green), логікою accordion для Hero / Action Card / Insights.
- **Чи можливо без поломки:** **так**. Це лише специфікація для розробки; сама по собі не змінює ні бекенд, ні поточну структуру. Розробка за Blueprint може бути поетапною і за флагом.

## 4. Ризики і як їх уникати

| Ризик | Як уникати |
|------|------------|
| Старі клієнти отримують нові поля і «ламаються» | Не додавати обов’язкові поля в існуючі контракти; Behavioral Summary — окремий ендпоінт або опційне поле в новому контракті. |
| Маячки викликають помилки на фронті | Фронт не блокує UI при 404/помилці мережі; відправка beacon у try/catch або sendBeacon з обробкою помилок. |
| Новий код бекенду ламає існуючі запити | Нові маршрути/таблиці не змінюють існуючі; feature flag вимикає новий шар повністю. |
| Зміна структури БД ламає міграції | Нові таблиці/колонки — окремі міграції, зворотні сумісні (nullable або з default). |

Якщо дотримуватися цих умов, **зміни можливі без поломки працюючої системи**.

## 6. Висновок

**Чи можливі запропоновані зміни без поломки працюючої системи?**  
**Так**, за умови:

1. **Не змінювати** існуючі контракти (SpecialistResponse, GET /api/specialists, GET /api/recommendations/my-stats тощо) — додавати лише нові ендпоінти або опційні об’єкти.
2. **Маячки** реалізувати так, щоб 404 або помилка мережі не ламали роботу сторінки.
3. **Новий функціонал** (Smart Layer, Action Card, Behavioral) вмикати через **Feature Flag**, поступово.
4. **TimeOnPage** на бекенді вводити адитивно (нова схема зберігання, новий прийом подій); до цього моменту матриця тригерів може використовувати лише «bookings=0» та інші вже наявні метрики.

Структуру проєкту та існуючий код у цій аналітиці **не змінювали** — лише перевірено репозиторій і сформульовано умови безпечної реалізації. Фінальний **Frontend Implementation Blueprint** (Hero, Action Card, Insights, джерела даних, стани Red/Orange/Green, accordion) можна готувати окремим документом і використовувати для верстки та логіки без ризику поламати поточну систему.
