# Інструкція: перевірка активності токена адміна та поновлення доступу до Cloud SQL

Документ описує **як адміну перевірити, чи ще активний його токен** (JWT платформи), та **як технічно поновлювати доступ (токен/облікові дані) до бази Google Cloud SQL** (інстанс `healer-nexus-db`). Код не змінюється — лише використання API та інструментів GCP.

---

## Частина 1. Токен адміна на платформі Healer (JWT)

### Як зрозуміти, чи ще активний access token

- **Спосіб 1 (рекомендований):** викликати **GET** `/api/auth/me` з заголовком  
  `Authorization: Bearer <ваш_access_token>`.
  - Якщо відповідь **200** і в тілі є дані користувача (id, email, role тощо) — токен **активний**.
  - Якщо відповідь **401** (Not authenticated / Invalid or expired token) — access token **протермінований або невалідний**.

- **Спосіб 2:** викликати будь-який захищений ендпоінт (наприклад **GET** `/api/admin/users`) з тим самим заголовком. 200 — токен активний, 401 — ні.

Термін дії (за замовчуванням у конфігурації):
- **Access token:** 30 хвилин.
- **Refresh token:** 7 днів (і зберігається в БД; його можна відкликати через logout).

Якщо access token протермінований, але є збережений **refresh token**, його можна поновлювати (див. нижче).

### Як поновлювати токени (без повторного логіну)

1. Відправити **POST** `/api/auth/refresh` з тілом (JSON):  
   `{"refresh_token": "<ваш_збережений_refresh_token>"}`.
2. У відповіді при успіху (200) прийдуть нові **access_token** та **refresh_token**; термін дії refresh знову 7 днів.
3. Далі використовувати новий **access_token** у заголовку `Authorization: Bearer <новий_access_token>` для викликів API.

Якщо відповідь 401 — refresh token невалідний або вже відкликаний (наприклад після logout). Тоді потрібно знову увійти: **POST** `/api/auth/login` (email + password) і зберегти нову пару access + refresh.

**Підсумок для адміна:** перевірка активності — **GET** `/api/auth/me` з поточним access token; поновлення — **POST** `/api/auth/refresh` з refresh token.

## Частина 2. Поновлення доступу до Cloud SQL (healer-nexus-db)

Інстанс бази в Google Cloud: [Cloud SQL — healer-nexus-db](https://console.cloud.google.com/sql/instances/healer-nexus-db/overview).  
Підключення з проєкту Healer використовує змінну середовища **CLOUD_SQL_CONNECTION_NAME** (формат `проєкт:регіон:інстанс`) та **DATABASE_URL** для PostgreSQL. Нижче — як технічно «поновлювати» доступ (облікові дані), без змін у коді.

### 2.1. Якщо підключаєтесь локально (Cloud SQL Auth Proxy або інструменти GCP)

Для локального доступу до Cloud SQL зазвичай використовуються **Application Default Credentials (ADC)**. Їх потрібно періодично оновлювати (після закінчення терміну дії або зміни облікового запису).

**Поновлення ADC (один раз або при 401 від GCP):**

1. Встановити [Google Cloud SDK (gcloud)](https://cloud.google.com/sdk/docs/install), якщо ще не встановлено.
2. У терміналі виконати:
   ```bash
   gcloud auth application-default login
   ```
3. Відкриється браузер для входу в Google-акаунт; після успішного входу облікові дані збережуться локально (файл у профілі користувача). Ці дані використовуються Cloud SQL Auth Proxy та іншими клієнтами GCP.

Після цього запуск **Cloud SQL Auth Proxy** з connection name вашого інстансу (наприклад `проєкт:регіон:healer-nexus-db`) знову буде використовувати актуальні облікові дані. Окремо «токен» для проксі оновлювати не потрібно — проксі сам використовує ADC.

### 2.2. Якщо додаток працює на Cloud Run (або іншому сервісі GCP)

На Cloud Run підключення до Cloud SQL йде через Unix-сокет (`/cloudsql/<CLOUD_SQL_CONNECTION_NAME>`). Доступ до БД використовує **сервісний акаунт**, під яким крутиться сервіс.

- **«Поновлення» тут** — це не щоденний refresh токена, а переконатися, що:
  1. У сервісному акаунті є роль доступу до Cloud SQL (наприклад **Cloud SQL Client**).
  2. Якщо ви змінили пароль користувача БД або сам інстанс — оновити змінні середовища сервісу (наприклад **DATABASE_URL**) в Cloud Run (редагування сервісу → змінні та секрети) і перезапустити/задеплоїти сервіс.

Окремий «токен» для Cloud SQL у цій схемі зазвичай не використовується: авторизація через IAM і сервісний акаунт. Якщо адмін має доступ лише до консолі GCP, достатньо перевірити права сервісного акаунта в [IAM](https://console.cloud.google.com/iam-admin) та налаштування інстансу в [Cloud SQL](https://console.cloud.google.com/sql/instances/healer-nexus-db/overview).

### 2.3. Пароль користувача бази даних (PostgreSQL)

Якщо **DATABASE_URL** містить пароль користувача БД і ви його змінили в інтерфейсі Cloud SQL:

1. Відкрити [Cloud SQL — healer-nexus-db → Users](https://console.cloud.google.com/sql/instances/healer-nexus-db/users) (або еквівалентний розділ для користувачів БД).
2. Змінити пароль потрібного користувача.
3. Оновити **DATABASE_URL** у середовищі, де запущений бекенд (файл `.env` локально або змінні/секрети в Cloud Run), щоб у URL був новий пароль.
4. Перезапустити додаток (локально або повторно задеплоїти/перезапустити сервіс на Cloud Run).

Це і є «поновлення» доступу в сенсі нового пароля: оновили пароль у GCP і той самий новий пароль вказали в **DATABASE_URL**.

### 2.4. Короткий чеклист для healer-nexus-db

| Ситуація | Дія |
|---------|-----|
| Локальний доступ через Cloud SQL Proxy / gcloud | Виконати `gcloud auth application-default login` для оновлення ADC. |
| Сервіс на Cloud Run не може підключитися до БД | Перевірити роль сервісного акаунта (Cloud SQL Client) та CLOUD_SQL_CONNECTION_NAME і DATABASE_URL у конфігурації сервісу. |
| Змінили пароль користувача PostgreSQL в GCP | Оновити пароль у DATABASE_URL (.env або змінні Cloud Run) і перезапустити додаток. |
| Переглянути інстанс у консолі | Відкрити [Cloud SQL — healer-nexus-db](https://console.cloud.google.com/sql/instances/healer-nexus-db/overview). |

## Довідка по API (токени платформи)

| Дія | Метод | URL | Примітка |
|-----|--------|-----|----------|
| Перевірити, чи активний access token | GET | `/api/auth/me` | Заголовок: `Authorization: Bearer <access_token>`. 200 — активний. |
| Отримати нову пару access + refresh | POST | `/api/auth/refresh` | Тіло: `{"refresh_token": "..."}`. |
| Увійти знову (новий access + refresh) | POST | `/api/auth/login` | Тіло: `{"email": "...", "password": "..."}`. |

Код проєкту в цих інструкціях не змінюється.
